# 🚀 Генеральная доктрина `MAKE_OP`: Полный Свод Знаний

### I. Первоисточник: Где брать Истину?
Сигнатуры — наше всё. **Никакой отсебятины.** Вот проверенный путь поиска:
1.  **YAML-конфиги Op-Plugin (`op_plugin_functions.yaml`)**:
    *   Ищи `func: npu_имя_операции(...) -> ...`.
    *   Здесь ты найдёшь **каноничные типы аргументов** (`Tensor`, `Scalar`, `bool`, `int`) и значения по умолчанию. Это твой главный ориентир.
2.  **JSON-схемы (`torch_npu_OpApi_schema_all.json`)**:
    *   Ищи `"func: npu_имя_операции..."`. Ключевое поле `exec: aclnnXxxYyy` даёт тебе **точное имя aclnn-функции**, которую нужно реализовать.
3.  **Заголовочники CANN (`*.h`)**:
    *   Итоговая инстанция: `ACLNN_API aclnnStatus aclnnCatGetWorkspaceSize(...)`. Копируем **1-в-1**, соблюдая порядок аргументов. Пробелы перед `*` и `&` удаляем.

### II. Анатомия aclnn: Двухтактный Двигатель
Помни: aclnn-операция — это две функции, а не одна:
1.  **`GetWorkspaceSize`**: Вызывается первой. `torch_npu` ожидает, что мы здесь **создадим Executor** и запросим временную память (мы всегда просим `0`).
2.  **Основная функция (`DEFINE_ACLNN_OP`)**: Вызывается второй. Получает созданный Executor и выполняет код из `MAKE_OP`.

**Ключевое Правило:** Сигнатуру `GetWorkspaceSize` **менять нельзя**. Если в ней нет выходного тензора (как `gradInput` в старых версиях `GluBackward`), то мы не можем использовать `MAKE_OP` и оставляем операцию как `UNIMPLEMENTED`.

### III. Синтаксис `MAKE_OP` и Магия Типов

**Базовый шаблон:**
```cpp
MAKE_OP(aclnnИмяОперации(
    [модификатор] ТипАргумента имя_аргумента,
    ...
    uint64_t* workspaceSize, aclOpExecutor** executor  // Всегда в конце!
) {
    // Твоё тело. Пиши чисто, без exec->.
})
```

#### Таблица Конвертации Типов (Что видишь в YAML → Что получаешь в теле MAKE_OP)

| Тип в YAML / Сигнатуре | Аннотация | Как объявлено в теле MAKE_OP | Примечание |
| :--- | :--- | :--- | :--- |
| `Tensor` (вход) | (нет) | `at::Tensor` | Автоматически загружается через `LOAD_TENSOR`. |
| `Tensor` (выход) | `out` | `at::Tensor` | Загружается. Логируется в `end()`. |
| `Tensor` (выход с resize) | `sync` | `at::Tensor` | Загружается. Логируется в `end()`. **После операции вызывается `SYNC_AFTER_MUTATION`.** |
| **`Tensor` (выход с `store`)** | **(особый)** | **Прямой доступ к `aclTensor*`** | **Единственный случай, где мы пишем `exec->buffer->store(...)`.** |
| `Scalar` (вход) | (нет) | `const at::Scalar&` | Автоматически разыменовывается из `aclScalar`. |
| `Scalar` (выход, Inplace) | `out` | `const at::Tensor&` | Только для in-place скаляров (например, `InplaceAddV3`). |
| `IntArray` | (нет) | `const std::vector<int64_t>&` | Автоматически разыменовывается. |
| `FloatArray` | (нет) | `const std::vector<float>&` | ... |
| `BoolArray` | (нет) | `const std::vector<uint8_t>&` | ... |
| `TensorList` | (нет) | `const at::TensorList&` | Автоматически разыменовывается. |
| `ScalarList` | (нет) | `const std::vector<const aclScalar*>&` | ... |
| `int64_t` / `double` / `bool` | (нет) | Значение (копия) | Генератор заменяет `имя` на `exec->имя`. |

### IV. Святая Троица Управления Памятью: `copy_` vs `sync` vs `store`

*   **`copy_`**: Твоя основная рабочая лошадка. Используй **всегда**, когда `torch_npu` уже выделил выходной тензор **правильного размера** и его не надо менять. (`out.copy_(result);`)

*   **`sync`**: Используется, когда форма выхода меняется, **но** `torch_npu` выделил **достаточно большой** буфер (обычно максимально возможный, как для `Nonzero`). Мы просто обрезаем размер через `SYNC_AFTER_MUTATION`. Пример:
    ```cpp
    // Форма меняется, но буфер выделен с запасом.
    at::nonzero_out(out, self); // out.resize_ внутри не требуется
    // SYNC_AFTER_MUTATION обновит desc->dims
    ```

*   **`store`**: Тяжёлая артиллерия. Только для случаев, когда `torch_npu` понятия не имеет о финальном размере и выделяет **нулевой буфер**. Мы **полностью заменяем** буфер и метаданные.
    ```cpp
    // aclnnLogSigmoidForward – уникальный случай, где буфер выхода пуст.
    auto res_buf = at::empty_like(self);
    at::log_sigmoid_forward_out(out, res_buf, self);
    exec->buffer->store(res_buf); // <-- Прямой доступ к aclTensor
    ```

### V. Исключение из Правил: Прямой Доступ к `exec->`
Это твой единственный козырь для обхода автоматической загрузки тензоров. Используй, когда тебе нужен **не ATen-тензор, а сам объект `aclTensor`** (чтобы вызвать его метод `store`).
*   **Правильно:** `exec->buffer->store(...)`
*   **Неправильно:** `exec->from`, `exec->self` (генератор уже превратил их в `at::Tensor`).

### VI. Кодогенератор: Что он делает с твоим `MAKE_OP`
1.  Парсит сигнатуру, определяет типы и модификаторы (`out`, `sync`).
2.  Генерирует `struct Исполнитель` с полями для всех аргументов.
3.  Создаёт `GetWorkspaceSize`, где заполняет эту структуру.
4.  В `DEFINE_ACLNN_OP` вставляет:
    *   `LOAD_TENSOR` для всех `aclTensor*`.
    *   `const at::Scalar&` для `aclScalar*`.
    *   `const auto&` для списков (`aclTensorList*`, `aclIntArray*`).
    *   Замену `имя_аргумента` на `exec->имя_аргумента` для примитивов.
    *   Вызовы `SYNC_AFTER_MUTATION` для всех аргументов с модификатором `sync`.
    *   Обработку исключений и логирование.

### VII. Процесс Создания Новой Операции (Чек-лист)
1.  **Найди сигнатуру** в YAML/JSON/H-файлах.
2.  **Напиши `MAKE_OP`**, следуя синтаксису и таблице типов.
3.  **Определись с памятью**: `copy_` (по умолчанию), `sync` (если форма меняется), `store` (если буфер нулевой).
4.  **Проверь ATen-функцию**: Если есть прямой аналог (`at::add_out`, `at::gelu`), используй его. Если нет — ручная реализация.
5.  **Запусти кодогенератор**: `not_opapi_gen.py` превратит твой `MAKE_OP` в горы корректного C++ кода.
6.  **Собери и тестируй**: `build_not_npu && pytest`.
